# aisinger · Design Brief

> 整体前端的设计章程。本文件为单一可信来源；UI 决策有冲突时以此为准。

---

## 1. Feature Summary

aisinger 是一个**本地 AI 歌手复刻工作台**：用户上传 1–5 段样本 → 系统提取音色 → 用该音色把任意歌曲翻唱出来（也支持仅文本说话）。

目标用户是**单个创作者**在自己 Mac 上使用，不存在多人协作、云同步、账号体系。

成功定义：从打开应用到拿到第一首 AI 翻唱，无需查文档、无需进终端。

---

## 2. Primary User Action

**选音色 + 选曲目 → 一键合成**。

整个 UI 应该让这件事的物理距离最短：默认落地页就是「合成」，其他都是支撑。

---

## 3. Design Direction

### 风格定位

参考 **Linear** 的工程美学，但**更克制**：

| 维度 | 取值 |
|---|---|
| 色调 | 深色为主（`#08090a` 起步），单一暗色主题。**不做亮色版**。 |
| 强调色 | Linear 紫 `#5e6ad2` / `#8b95ff`。**全应用唯一强调色**，不引入第二种 hue。 |
| 字体 | Inter Variable，正文 14px，`letter-spacing: -0.011em`，开启 `ss01` `cv11`。等宽字 `Berkeley Mono` 仅用于时间戳、id、数字读数。 |
| 边框 | 发丝 `rgba(255,255,255,0.07)`。**禁止阴影分层**；只用 1px 边 + 微差底色。 |
| 圆角 | 6 / 8 / 10 三档。按钮 6，卡片 8，模态 10。**不要 16+ 大圆角**。 |
| 进度条 | 2–3px 高，**单色无渐变**。 |
| 状态指示 | 仅 2 色：绿点 = 就绪 / 蓝色脉冲 = 训练中。**不引入第三种 status color**。 |

### 气质

- **像开发者工具，不像消费品**。冷静、精确，不卖萌。
- **密度由功能驱动**：列表页极简，复杂表单（如新建音色）允许更密。
- **数字是核心读物**：进度百分比、ETA、时长、loss 值都应该用等宽字 + tnum，让对齐自动成立。

### Anti-Goals (绝不要)

- ❌ 多种 status color（红/橙/黄/绿堆叠）
- ❌ 卡片阴影、玻璃拟态、Neumorphism
- ❌ 大圆角 + 粗描边的卡通风
- ❌ Emoji 装饰、tier 大徽章、icon 满天飞
- ❌ 多余的元数据展示（kHz、LU、艺人详情等开发者噪音泄漏给用户）
- ❌ 弹跳、橡皮、长动画 — 所有过渡 120–200ms，`ease-out-quart` 收尾

## 4. Layout Strategy

### 主框架

```
┌──────────┬──────────────────────────────────┐
│ Sidebar  │  Main                            │
│  200px   │   max 760px center, padded 56px  │
│          │                                  │
│ logo     │   h1 24px                        │
│ 合成 ●   │   …content…                      │
│ 音色库   │                                  │
│ 曲目库   │                                  │
│ AI 歌曲库│                                  │
│          │                                  │
│ mps·ready│                                  │
└──────────┴──────────────────────────────────┘
```

**核心规则**：

1. **侧栏永远 200px**，永远只有 4 个 nav 项 + 设备状态 footer。**不加二级菜单、不加运行卡、不加搜索**。
2. **主区域 max 760px 居中**，列表页使用全宽列表行（不带卡片包裹），表单页（新建音色）允许扩到 880px。
3. **没有顶 bar、没有 breadcrumb、没有 tab**。每页只有 h1 + 内容。
4. **没有右侧常驻面板**。详情用模态或行内展开。

### 信息层级

- **第一层** = 用户主操作（按钮、可选择的对象） — 文字 14px，强调色或纯白
- **第二层** = 行的副信息（艺人、时长、状态） — 12px，灰 `--fg-2`
- **第三层** = 元数据（id、文件大小、时间戳） — 11px 等宽，更浅 `--fg-3`

### 节奏

- 区块间距 24/32px
- 行内间距 8/12/16px 三档
- 不在同一密度尺度上跳级（避免 4 → 24 这种空旷感）

## 5. Key States

| 视图 | 默认 | 空 | 加载 | 错误 | 边缘 |
|---|---|---|---|---|---|
| 合成 | 两个 row + 灰按钮 | 音色库为空时引导"先建一个音色 →" | 合成中：按钮变 pulse + 文案"合成中…" | 失败：红字 inline 提示 + 重试按钮 | 选了 simple 档音色时第二个 row 自动变文本输入 |
| 音色库 | 列表 + 训练中行内 progress | 引导卡：上传 1–5 个样本开始 | — | 训练失败：行变红，附"重试"链接 | 训练中可点击展开 7 阶段 |
| 曲目库 | 列表 + 上传按钮 | "上传你想用 AI 翻唱的歌" | 上传中：行内进度条 | 格式不支持：行内红色提示 | 分离失败回退使用原音频 |
| AI 歌曲库 | 列表 | "去合成页生成第一首吧" | — | — | 项目可收藏、评分 |
| 新建音色 | 4 段表单 + sticky 底栏 | — | 提交后跳音色库展示训练进度 | 样本不达标：sticky 底栏变红 + 具体说明 | 不同档位提示不同文案 |

### 训练状态：唯一的"复杂状态"

- 训练中的音色，在音色库里整行可点击**就地展开**为 7 阶段进度
- 各阶段：人声分离 / 切片归一化 / RVC preprocess / f0 / hubert / 训练主网络 / 索引
- 实时进度由 WebSocket 推送（待实现）
- **不开右侧面板、不跳新页**。展开 = 行高度从 56px 变为约 360px，平滑动画

## 6. Interaction Model

### 全局规则

- **键盘**：`Esc` 关一切模态。**不引入 ⌘K**（MVP 不需要）。
- **路由**：单 SPA，hash 路由（`#/synth` 等），允许后退键。
- **反馈**：所有操作 < 150ms 给出可见变化（按钮 active state、加载 pulse、loading skeleton）。
- **不弹通知**。完成事件用就地状态变更体现（如训练完成 → 行从蓝 pulse 变绿点 + 时间"刚刚"）。
- **不引入 toast / snackbar 等浮层**。

### 合成页

- 左右两个大 row（音色 / 曲目）→ 点开模态选择
- 选完后两 row 显示选中态
- 「更多参数」折叠默认关
- 「开始合成」按钮在两个 row 都选好之前为 disabled 灰态
- 合成产生后下方插入 player + meta；持续点合成可堆叠多个结果

### 音色库

- 静态音色：一行展示 name + 「使用」按钮
- 训练中音色：整行点击展开为完整进度面板
- 「＋ 新建音色」按钮在右上角 → 跳整页新建流程（不是模态）

### 新建音色（唯一允许复杂的页面）

- 4 段 section 顺序：档位 → 命名+样本 → 预处理预览 → 训练参数
- **不是 wizard**（不强制 Next/Back），整页可滚动
- 顶部 chip 步骤指示器仅是路标，不可点跳
- 底部 sticky bar 实时显示 "中等档 · 我的声音 · 5 样本 · 20 epochs · 预计 28 分钟" + 取消 / 开始训练
- 提交后跳音色库，新建项 = 行最顶 + 训练中状态

### 曲目库

- 一行展示 name + 艺人 + 时长 + ▶
- 上传后台分离人声（Demucs，约 3 min/首），分离中行内 progress

### AI 歌曲库

- 一行展示 name + 「音色 · 时间」 + ▶
- 点击行内展开 player（不是模态）

## 7. Content Requirements

### Voice

- 文案规范：用户自己起名，避免"模型 v1"这种工程黑话
- 默认占位："例：我的声音"
- 训练时长描述："预计 28 分钟"（用区间词"约/预计"，**不要**伪精确"00:28:14"展示给用户主动看的位置，仅在悬浮/详情里用 etaStr）

### 状态文案

| 状态 | 用户文案 | 工程文案 |
|---|---|---|
| 训练中 | "训练中 · 12%" | `running` |
| 就绪 | "就绪" | `done` |
| 失败 | "失败 · 重试" | `failed` |
| 上传中 | "上传 42%" | `uploading` |
| 分离中 | "分离人声中" | `separating` |

### 空状态文案

- 音色库空：「**还没有音色**\n上传 1–5 段歌唱或说话音频，开始训练」 + 主 CTA「＋ 新建音色」
- 曲目库空：「**曲目库为空**\n上传你想 AI 翻唱的歌曲」 + 主 CTA「＋ 上传曲目」
- AI 歌曲库空：「**还没有作品**\n去合成页生成第一首吧」 + 次级 CTA「→ 去合成」

### 错误文案

- 永远说**这是什么 + 怎么办**，不要堆栈
  - ✅「文件太大了（最大 30 MB）。试试切短一点。」
  - ❌「Error: file size exceeds 30MB limit (got 35MB)」

### 微文案禁忌

- 不要"亲""哦""啦"
- 不要"AI 一下""魔法""魔术"
- 不要"完美""惊艳"等夸张评价词
- 错误页不要"Oops"

## 9. Open Questions

实施时由开发者临场决定：

1. **WebSocket 协议形式** — 直接走 FastAPI WS 还是用 Server-Sent Events？训练进度是单向推送，SSE 可能更简单。**建议**：用 SSE。
2. **音色 / 曲目预览音频**是否做客户端波形分析（更精确）还是后端预生成静态 PNG（更快首屏）？**建议**：MVP 阶段直接用 `<audio>` + 假波形装饰，后续优化。
3. **训练失败的恢复路径**：保留中间产物允许"从某阶段继续"？还是失败即从头？**建议**：MVP 失败即从头，简化状态机。
4. **A/B 对比**当前不作为 nav 入口，需要明确入口：从 AI 歌曲库选两条「比较」？还是合成页结果区"和另一首对比"按钮？**建议**：从 AI 歌曲库行的菜单触发 modal。
5. **侧栏 footer "mps · ready"**：要不要可点击查看 GPU / 内存 / ffmpeg 状态？**建议**：MVP 不点击，只是显示。

## 决策日志（迭代过程沉淀）

| 决策 | 选择 | 反面方案 | 理由 |
|---|---|---|---|
| 设计语言 | Linear-ish 深色克制 | Notion 浅色友好 / Studio 重技术风 | 用户为单个开发者，深色 + 工程美学最贴 |
| Nav 项 | 4 个：合成 / 音色 / 曲目 / 作品 | 7 个（含主页/A/B/设置/任务） | 信息架构去层级，每个 nav 都直接是一个数据集合 |
| 默认页 | 合成 | 主页 dashboard | 合成是高频主动作 |
| 训练详情位置 | 行内 accordion | 右侧面板 / 独立页 / drawer | 持续状态但低注意力，行内最不打断 |
| 设置 | 不做独立页 | 侧栏底部 settings 入口 | MVP 阶段没什么可设置 |
| ⌘K 命令面板 | 不做 | 全局命令面板 | 4 个 nav 直接点比键盘命令快 |
| 通知系统 | 不做 toast | 右下角 toast / 顶部 banner | 完成事件就地状态变化即可 |
